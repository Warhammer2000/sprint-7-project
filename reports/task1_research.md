# Задание 1 — исследование моделей и инфраструктуры

**Кто заказчик и для кого решение.** Заказчик — QuantumForge Software
(финско‑эстонская продуктовая компания). Пользователи бота — разработчики,
саппорт, менеджеры/аналитики и новые сотрудники, которым нужен быстрый и точный
ответ по внутренней базе знаний (~18 000 Markdown, ~3 000 страниц Confluence,
~250 PDF). Ключевое ограничение: **часть документации конфиденциальна**
(ADR, политики, SOC 2), её нельзя бесконтрольно выносить в публичное облако.

## 1. LLM: локальные Hugging Face vs облачные OpenAI / YandexGPT

| Критерий | Локальные HF (Llama 3, Qwen, Saiga, Mistral) | OpenAI (GPT‑4o / 4o‑mini) | YandexGPT |
| --- | --- | --- | --- |
| Качество ответов (RU) | Среднее–высокое (зависит от размера) | Высокое | Высокое для русского |
| Скорость | Зависит от GPU; на CPU медленно | Низкая латентность, но сетевая | Низкая латентность в РФ |
| Стоимость владения | CAPEX на GPU + DevOps, но дальше «бесплатно» за токен | OPEX, оплата за токены | OPEX, оплата за токены |
| Развёртывание | Сложно (GPU, vLLM/TGI, квантование) | Тривиально (API‑ключ) | Тривиально (API‑ключ) |
| Конфиденциальность | **Данные не покидают периметр** | Данные уходят к вендору | Данные уходят к вендору (но в РФ‑контуре) |

**Вывод.** Для конфиденциального контура — локальная open‑source LLM (например
Llama 3 8B/Qwen2.5 в quantized‑формате через Ollama/vLLM). Для неконфиденциальных
сценариев допустимо облако (gpt‑4o‑mini как баланс цена/качество).
В коде это решено **подключаемым слоем LLM** (`ragbot/llm.py`: `offline` /
`openai` / `ollama`) — провайдер меняется одной переменной `.env`, без переписывания
пайплайна.

## 2. Эмбеддинги: локальные Sentence-Transformers vs OpenAI Embeddings

| Критерий | Sentence-Transformers (локально) | OpenAI `text-embedding-3` |
| --- | --- | --- |
| Скорость индексации | Высокая на GPU, приемлемая на CPU; данные локально | Сетевые вызовы батчами, лимиты API |
| Качество поиска | Хорошее; `paraphrase-multilingual-MiniLM-L12-v2` понимает русский | Очень высокое, но избыточно для домена |
| Стоимость владения | Бесплатно после установки (модель ≈ 120 МБ) | Оплата за токены при каждом (ре)индексировании |
| Конфиденциальность | **Тексты не уходят наружу** | Тексты уходят к вендору |
| Размерность | 384 (MiniLM) / 768 (mpnet, e5) | 1536 / 3072 |

**Вывод.** Для корпоративной базы с конфиденциальными данными —
**локальные Sentence-Transformers**. По умолчанию выбран
`paraphrase-multilingual-MiniLM-L12-v2` (384 dim): мультиязычный, лёгкий, быстрый.
Если нужен максимум качества по англоязычной части — `e5-base`/`mpnet-base-v2`.
В проекте эмбеддер тоже подключаемый (`ragbot/embeddings.py`): `st` для качества
и `tfidf` (LSA на scikit-learn) как сверхлёгкий офлайн‑фоллбэк без torch.

## 3. Векторная БД: ChromaDB vs FAISS

| Критерий | FAISS | ChromaDB |
| --- | --- | --- |
| Скорость поиска/индексации | Сверхбыстрый ANN in-process (C++) | Быстрый in-process, поверх SQLite/HNSW |
| Сложность внедрения | `pip install faiss-cpu`, 5 строк кода | `pip install chromadb`, тоже просто |
| Метаданные/фильтры | Нет «из коробки» (нужно вести рядом) | Есть фильтры по метаданным, persist «из коробки» |
| Удобство | Минимализм, полный контроль | «Pythonic», тесная интеграция с LangChain |
| Масштаб | Миллионы векторов в RAM | До десятков млн, дальше миграция |
| Стоимость владения | Бесплатно, on-prem | Бесплатно, on-prem |

> ChromaDB в уроках подробно не разбирался; по доп. исследованию — это
> встраиваемая БД с персистентностью и фильтрами по метаданным, удобная для PoC
> на LangChain, но при десятках млн векторов требует миграции (как и FAISS).

**Вывод.** Для этого проекта выбран **FAISS**: он рекомендован в уроке как
стартовый движок, не требует сервера, даёт максимальную скорость и полный
контроль; метаданные мы храним рядом (`meta.pkl`) и фильтруем сами. При росте до
десятков млн векторов или потребности в сетевом API/гео‑фильтрах — миграция на
**Qdrant** (один `docker run`, тот же код запросов).

## 4. Конфигурация сервера (CPU / RAM / GPU): 3–4 варианта

| Вариант | CPU | RAM | GPU | Кому подходит | Минусы |
| --- | --- | --- | --- | --- | --- |
| **A. CPU-only (бюджет)** | 8 vCPU | 16 ГБ | — | Эмбеддинги ST на CPU + облачная LLM; PoC и малый трафик | LLM локально не потянет; зависимость от облака |
| **B. CPU + облачная LLM (рекоменд.)** | 16 vCPU | 32 ГБ | — | Индекс + retrieval локально (FAISS+ST), генерация — облачный LLM по неконфиденц. данным | Конфиденц. тексты всё же уходят в облако для генерации |
| **C. 1×GPU mid (баланс конфиденциальности)** | 16 vCPU | 64 ГБ | 1×24 ГБ (L4/A10) | Полностью on-prem: локальная LLM 7–8B (quantized) + ST‑эмбеддинги | CAPEX на GPU, нужен MLOps |
| **D. Multi-GPU (масштаб/SLA)** | 32+ vCPU | 128+ ГБ | 2–4×A100/H100 | Высокий трафик, большие модели, низкая латентность | Дорого, избыточно для текущего объёма |

**Почему так (опора на урок про CPU/GPU):** эмбеддинги и FAISS‑поиск отлично
работают на **CPU** (последовательные/лёгкие операции, объём базы — сотни тысяч
чанков помещаются в RAM). **GPU** нужен только под локальную **генеративную LLM**
(трансформер с миллиардами параметров — параллельные вычисления).

---

## Итоговая рекомендация для QuantumForge

1. **Векторная БД — FAISS** (старт), миграция на Qdrant при масштабировании.
2. **Эмбеддинги — локальные Sentence-Transformers** `paraphrase-multilingual-MiniLM-L12-v2`
   (конфиденциальность + русский язык).
3. **LLM — гибрид:** локальная open‑source LLM (Ollama/vLLM) для конфиденциальных
   документов; облачный gpt‑4o‑mini/YandexGPT для неконфиденциальных — выбор через
   `.env`.
4. **Сервер — вариант C** (1×GPU 24 ГБ + 64 ГБ RAM) как баланс приватности,
   стоимости и качества; для пилота можно стартовать с **варианта B** (CPU +
   облачная LLM) и докупить GPU при переходе на конфиденциальный контур.
5. **Безопасность данных:** конфиденциальные тексты не покидают периметр
   (локальные эмбеддинги + локальная LLM), что снимает риски, отмеченные в кейсе
   (SOC 2, политики).

Эти решения заложены в архитектуру проекта: подключаемые слои LLM и эмбеддингов
позволяют переключать конфигурацию A→D без изменения RAG‑пайплайна.
